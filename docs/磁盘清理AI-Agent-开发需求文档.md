# 磁盘清理 AI Agent 开发需求文档

版本：V0.2

## 1. 产品命名建议

### 1.1 推荐名称
**DiskClaw**

命名理由：
- `Disk` 直接表达磁盘管理与空间治理场景。
- `Claw` 传达“精准抓取、识别、清除冗余文件”的动作感，利于塑造 AI Agent 的执行力形象。
- 英文名简短，适合产品名、项目名、域名、安装包名和品牌延展。

### 1.2 中文名称建议
**磁盘清理大虾**

中文定位语：
**像专家一样找出空间黑洞，像助手一样安全完成清理。**

### 1.3 备选名称
- **ClawClean**：偏工具感，直观但品牌辨识度略弱。
- **SpaceClaw**：更强调空间释放，适合面向普通用户。
- **DiskPilot**：更偏智能助手路线，但“清理”特征稍弱。
- **CleanOrbit**：更偏品牌化，适合做较完整产品矩阵。

## 2. 项目背景

用户在日常使用电脑时，会持续积累下载文件、安装包、日志、缓存、重复文件、大文件和无效残留。传统磁盘清理工具通常存在以下问题：
- 只提供扫描结果，不理解“为什么可以删”。
- 风险提示不足，用户害怕误删。
- 规则式清理能力强，但对复杂场景缺少解释与建议。
- 不具备持续学习用户偏好和清理习惯的能力。

因此，需要设计一款以 AI Agent 为核心的磁盘清理产品，通过“发现问题、解释原因、给出建议、执行清理、复盘优化”的闭环能力，帮助用户更安全、更高效地管理磁盘空间。

本产品的 Agent 核心推理、解释、对话和方案生成能力基于 LLM API 运行，因此需要提供标准化的第三方 LLM API 接入接口，支撑不同模型服务的切换、配置、容错和扩展。

## 3. 产品目标

### 3.1 核心目标
- 帮助用户快速识别磁盘空间占用异常和可清理项。
- 通过 AI 解释与风险分级，降低误删焦虑。
- 提供从扫描、分析、决策到执行的一体化清理体验。
- 支持持续化、个性化的磁盘空间治理。
- 建立稳定、可扩展的第三方 LLM API 接入能力，作为 Agent 运行基础。

### 3.2 业务目标
- 提升扫描到清理的转化率。
- 提升用户对“建议可信度”和“操作安全感”的感知。
- 降低误删导致的投诉和数据恢复成本。
- 为后续会员能力扩展预留空间，如深度分析、自动清理、企业策略管控等。

## 4. 目标用户

### 4.1 个人用户
- 电脑空间不足但不清楚该删什么。
- 想释放空间，但担心删错重要文件。
- 需要简单易懂的清理建议和一键处理能力。

### 4.2 进阶用户
- 希望查看更详细的空间分布、重复文件、历史版本和大文件。
- 需要更强的可控性，如白名单、黑名单、自定义规则。

### 4.3 企业/团队用户（后续扩展）
- 需要统一设备清理策略。
- 需要日志审计、规则下发和安全边界控制。

## 5. 产品定位

DiskClaw 不是单纯的“清理按钮”，而是一款具备感知、判断、解释和执行能力的磁盘治理 AI Agent。

产品定位可以概括为：
**一个以安全解释为核心的智能磁盘清理助手。**

## 6. 核心使用场景

### 6.1 空间告急
用户发现磁盘空间不足，希望快速知道“哪些内容最占空间、哪些能安全删除”。

### 6.2 周期清理
用户希望定期清理浏览器缓存、临时文件、安装残留、回收站等低风险内容。

### 6.3 大文件治理
用户想定位长期未使用、体积巨大、重复保存或无效备份的文件。

### 6.4 开发者/设计师设备治理
用户设备上存在构建缓存、依赖包、日志文件、导出包、中间产物等，需要更专业的分类和建议。

### 6.5 清理前咨询
用户不确定某个文件夹是否可以删除，希望先获得解释、影响评估和替代建议。

## 7. 产品范围

### 7.1 本期范围
- 本地磁盘扫描与空间分析
- 垃圾文件识别
- 大文件与重复文件识别
- AI 清理建议与自然语言解释
- 第三方 LLM API 接入与模型配置
- 风险分级与确认机制
- 一键清理与选择性清理
- 清理记录与恢复建议

### 7.2 暂不纳入本期

- 完整的数据恢复系统
- 跨设备统一知识库

## 8. 功能需求

### 8.1 首页总览

功能说明：
- 展示当前磁盘使用情况、已用空间、可释放空间估算值。
- 展示高优先级问题，如“系统盘不足 10%”“发现超大视频包”“检测到重复安装包”等。
- 展示 AI Agent 的一句话摘要建议。

关键需求：
- 支持按磁盘分区展示空间概况。
- 支持展示本次扫描时间和上次清理收益。
- 支持用户点击进入详细分析页。

### 8.2 磁盘扫描

功能说明：
- 自动扫描本地文件系统，识别可清理对象和空间热点。

扫描对象：
- 临时文件
- 系统缓存
- 应用缓存
- 下载目录冗余文件
- 回收站
- 日志文件
- 安装包和解压残留
- 大文件
- 重复文件
- 长期未访问文件

关键需求：
- 支持快速扫描与深度扫描两种模式。
- 支持暂停、继续和取消扫描。
- 支持扫描过程中的实时结果回传。
- 支持按目录、文件类型、文件大小、修改时间等维度聚合。

### 8.3 AI 智能分析

功能说明：
- 基于扫描结果，由 Agent 通过 LLM API 输出清理建议、原因解释、风险评级和预估收益。

关键需求：
- 用自然语言告诉用户“为什么建议删除”。
- 对每个建议项说明占用空间、最近使用时间、删除影响范围。
- 支持把多个扫描结果整合成一个“清理方案”。
- 支持根据用户提问做追问式解释，如“这个文件夹删了会怎样？”。
- 当 LLM API 不可用时，系统应明确提示当前能力受限，并可降级到本地规则扫描结果展示。

示例输出：
- 检测到 `Downloads` 目录下存在 12 个超过 1GB 且 90 天未访问的安装包，预计可释放 18.4GB，风险等级为低，建议优先清理。

### 8.4 风险分级与安全防护

风险等级建议：
- 低风险：缓存、临时文件、回收站、明显重复项
- 中风险：下载目录安装包、旧日志、历史导出文件
- 高风险：系统目录、应用运行目录、用户文档、桌面文件

关键需求：
- 高风险文件默认不可一键删除，必须二次确认。
- 对系统关键目录建立硬性保护规则。
- 支持白名单和排除路径配置。
- 清理前展示影响摘要。
- 对无法确定安全性的对象标记为“建议人工确认”。

### 8.5 执行清理

功能说明：
- 按 AI 方案或用户自选方案执行清理动作。

执行方式：
- 删除到回收站
- 永久删除
- 移动到指定归档目录
- 跳过处理

关键需求：
- 默认优先删除到回收站。
- 清理任务需支持进度展示、失败重试和中断恢复。
- 对每个动作写入操作日志。
- 完成后输出清理报告。

### 8.6 对话式交互

功能说明：
- 用户可以通过自然语言直接发起问题和命令。

典型指令：
- “帮我找出可以安全删除的大文件”
- “为什么推荐清理这个目录？”
- “只清理浏览器缓存和临时文件”
- “以后不要扫描 D 盘的工作资料目录”

关键需求：
- 支持问答、建议、确认、执行四类对话。
- 支持上下文记忆，如记住用户排除目录偏好。
- 对敏感操作必须进行结构化确认，不能只依赖一句自然语言就直接执行。

### 8.7 历史记录与报告

功能说明：
- 记录每次扫描、建议和清理结果，支持复盘与趋势分析。

关键需求：
- 展示释放空间总量、最近 7/30 天清理趋势。
- 记录用户采纳率、跳过原因和失败项。
- 支持导出清理报告。

### 8.8 第三方 LLM API 接入与模型管理

功能说明：
- 提供标准化模型接入层，使 Agent 的对话、解释、建议生成和任务编排能力能够基于第三方 LLM API 运行。

关键需求：
- 支持配置第三方 LLM API 的 `Base URL`、`API Key`、模型名称、超时时间和重试策略。
- 支持接入多家模型服务商，至少具备统一适配层，避免业务逻辑直接耦合单一厂商。
- 支持 OpenAI 兼容接口风格，并预留厂商原生协议扩展能力。
- 支持模型连通性测试，在用户保存配置前可验证 API 是否可用。
- 支持按能力选择模型，如对话模型、推理模型、总结模型使用不同配置。
- 支持模型切换、默认模型设置和后续扩展多模型路由策略。
- 支持对 API 调用失败、超时、限流、鉴权失败进行明确错误提示和自动重试。
- 支持记录调用日志的结构化元数据，如请求时间、耗时、模型名称、状态码，但默认不记录敏感密钥和完整提示词原文。
- 支持 API Key 本地加密存储，界面展示时进行脱敏处理。
- 支持在设置页查看当前 Agent 所依赖的模型服务状态。

## 9. Agent 能力设计

### 9.1 Agent 角色定义
- **扫描 Agent**：负责文件系统扫描、分类和基础统计。
- **分析 Agent**：负责清理候选项聚合、优先级排序和风险判断。
- **解释 Agent**：负责把技术结果翻译成用户易理解的建议。
- **执行 Agent**：负责按确认策略执行清理动作并回传结果。
- **模型接入层**：负责第三方 LLM API 的适配、鉴权、路由、重试和统一响应封装。

### 9.2 Agent 工作流
1. 收集磁盘与文件元数据
2. 进行规则识别和特征提取
3. 生成候选清理项
4. 通过 LLM API 生成分析结果、解释文本和清理方案
5. 进行风险评估与优先级排序
6. 输出建议和解释
7. 用户确认后执行
8. 生成报告并更新用户偏好

### 9.3 决策原则
- 安全优先于清理收益
- 可解释性优先于黑盒推荐
- 用户可控优先于全自动执行
- 默认保守，逐步建立信任

## 10. 系统架构建议

### 10.1 模块划分
- 客户端 UI 层
- 本地扫描引擎
- 规则引擎
- Agent 编排层
- LLM API 适配与路由层
- 安全策略层
- 任务调度与日志模块
- 本地配置与用户偏好存储

### 10.2 建议架构原则
- 文件扫描与清理动作优先在本地执行。
- 上传云端的数据应最小化，原则上只传结构化元数据，不上传文件内容。
- Agent 的核心推理、对话和解释基于第三方 LLM API 运行，但不直接绕过安全策略操作文件。
- 业务层不得直接耦合具体模型厂商，所有模型调用统一经过 LLM API 适配层。
- LLM API 适配层需支持鉴权、超时、重试、限流、熔断和统一错误处理。
- 所有实际删除动作必须经过本地权限校验和策略网关。

## 11. 非功能需求

### 11.1 性能
- 快速扫描首屏结果返回时间不超过 5 秒。
- 常规扫描任务在用户可接受时长内完成，并支持后台运行。
- 在大目录扫描时保持 UI 可交互。

### 11.2 安全
- 默认不处理系统关键目录。
- 所有高风险操作需二次确认。
- 提供操作日志和异常追踪能力。

### 11.3 隐私
- 默认不上传文件原文内容。
- 用户可查看 Agent 使用了哪些本地数据做判断。
- 支持关闭云端分析能力，仅使用本地规则模式。
- 第三方 LLM API 所需的密钥、本地配置和调用记录应进行最小化存储与脱敏处理。

### 11.4 可维护性
- 规则库可配置、可扩展。
- Agent 提示词、策略模板、风险规则应模块化管理。
- 关键能力支持灰度发布和 A/B 测试。
- 模型适配器应插件化设计，便于新增或替换第三方 LLM 服务。

### 11.5 可用性
- 当第三方 LLM API 不可用时，系统应提供明确降级提示，不得静默失败。
- 单次模型调用失败后应支持重试与备用模型切换策略。
- 设置页应提供模型连通性检测和最近一次失败原因展示。

## 12. 平台与技术要求

### 12.1 首期平台建议
- Windows 优先

原因：
- 磁盘清理需求更高频
- 用户对安装包、缓存、日志、临时文件治理诉求更明确
- 便于形成第一版规则库与产品闭环

### 12.2 技术选型建议
- 客户端：Electron 或 Tauri
- 本地扫描引擎：Rust / Go / C#
- Agent 编排：Node.js / Python
- 数据存储：SQLite
- LLM 接入：第三方 LLM API 统一适配层 + 可配置模型网关 + 本地降级规则引擎

### 12.3 LLM API 接口设计建议
- 提供统一的模型调用接口，屏蔽不同供应商请求格式差异。
- 建议接口能力至少包括：`chat`、`reason`、`summarize`、`healthcheck`。
- 建议支持流式输出，便于展示 Agent 实时思考和建议生成过程。
- 建议支持请求级参数控制，如 `model`、`temperature`、`max_tokens`、`timeout`。
- 建议支持统一错误码映射，避免上层 UI 和业务逻辑感知厂商差异。
- 建议支持后续扩展私有网关、代理服务和企业自建模型中转层。

## 13. 权限与合规要求

- 首次执行清理前明确告知用户权限范围。
- 对涉及系统目录、管理员权限的动作单独授权。
- 敏感目录应提供不可删除保护。
- 清理建议必须避免诱导性表达，如“完全无风险”，应使用审慎表达。
- 第三方 LLM API 密钥需安全存储，不得明文写入日志、埋点或异常上报。
- 当调用第三方模型服务时，应向用户清晰说明上传的是哪些元数据以及用途。

## 15. MVP 范围建议

MVP 建议先聚焦以下能力：
- 系统盘空间扫描
- 临时文件、缓存、回收站识别
- 大文件识别
- AI 清理建议说明
- 第三方 LLM API 配置页与连通性测试
- 风险分级
- 一键清理与清理报告

MVP 暂不做：
- 重复文件的高级语义识别
- 自动化定时清理
- 企业策略管理
- 多 Agent 可视化编排
- 复杂多模型协同编排策略

### 里程碑 1：原型验证

## 17. 验收标准

- 用户能在 3 步内完成一次基础扫描和清理。
- 系统能明确区分低、中、高风险清理项。
- 每个 AI 建议项都有可读解释和空间收益说明。
- 高风险清理项不能被无确认直接执行。
- 清理完成后可查看结果报告和操作记录。
- 用户可在设置页完成第三方 LLM API 配置、连通性校验和模型切换。
- 当 LLM API 失败时，系统能返回明确错误信息或降级结果。

## 18. 风险与待确认项

### 18.1 主要风险
- 不同应用缓存目录规则复杂，误判风险较高。
- 用户对“AI 建议删除”天然敏感，解释质量要求高。
- 大规模扫描性能和资源占用需要重点优化。
- 第三方 LLM API 的可用性、成本、延迟和限流策略会直接影响 Agent 体验。

### 18.2 待确认问题
- 是否需要首期支持 macOS？
- 是否允许“自动清理模式”进入 MVP？
- 首期是否仅支持 OpenAI 兼容接口，还是同时支持厂商原生协议？
- 是否需要支持私有部署的模型网关或企业内部代理？
- 是否需要面向开发者提供高级清理规则页？

## 19. 一句话产品定义

**DiskClaw 是一款基于第三方 LLM API 运行、以安全解释为核心、能够发现空间问题并协助用户完成清理执行的磁盘治理 AI Agent。**
