# 磁盘清理大虾 0.1.1 开发排期

版本目标：0.1.1  
日期：2026-05-15

## 一、排期目标

`0.1.1` 定位为 `0.1.0` 的稳定性与体验修复版本，目标是在不大改主功能的前提下，提升发布质量、异常提示和细节一致性。

## 二、排期假设

- 1 名前端 / 桌面开发
- 1 名兼任后端 / 工程支持
- 1 名测试或由开发兼任验收
- 总工期：5 个工作日

## 三、工作拆分

### Day 1：问题梳理与修复准备

- 收拢 `0.1.1` 任务清单中的 P0 项
- 确认 `verify:release`、`smoke:desktop`、`smoke:e2e` 基线通过
- 建立修复分支与问题清单

交付物：

- 修复范围确认
- 当日问题清单

### Day 2：P0 稳定性修复

- 处理 Electron 脚本中的 `DEP0190` 相关告警
- 补任务详情 / 报告详情的摘要兜底显示
- 优化恢复流程异常提示

交付物：

- 第一批修复提交
- 自测通过

### Day 3：P1 体验打磨

- 隔离区增加“全选 / 清空选择”
- 任务与报告标签表现统一
- 设置页与执行区反馈提示统一

交付物：

- 第二批修复提交
- 页面体验对齐

### Day 4：测试补强与回归

- 为新增交互补自动化测试
- 回归任务列表、报告列表、隔离区恢复、设置页默认值
- 再跑一次发布验证链

交付物：

- 测试补充完成
- `npm run verify:release` 通过

### Day 5：版本收口与重新打包

- 更新 `0.1.1` 版本说明
- 重打安装包
- 更新验收记录与发布检查清单
- 形成最终发布目录

交付物：

- `0.1.1` 安装包
- `0.1.1` 发布说明
- 最终验收记录

## 四、优先级建议

### 必须完成

- 所有 P0 项
- 自动验证链路通过
- 安装包可重新构建

### 建议完成

- 至少 2 项 P1 体验优化
- 测试覆盖新增交互

## 五、发布门槛

满足以下条件后再发布 `0.1.1`：

1. 自动验证全部通过
2. 人工验收关键路径通过
3. 安装包重打成功
4. 版本说明与上传物料同步更新
